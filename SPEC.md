# Snapmeal — MVP仕様書

> 「冷蔵庫を撮るだけで、今日の献立が決まる」

---

## 1. プロダクト定義

### 本質

これは**レシピ検索アプリではない**。

ユーザーが欲しいのは「今日これを作ればいい」と決めてもらう体験。  
朝・昼・夜問わず、食事の意思決定を丸ごとAIに委ねるオペレーションツール。

### アプリ名・ブランド

| 項目 | 決定内容 |
|---|---|
| アプリ名 | **Snapmeal** |
| コンセプト | Snap（撮る）+ Meal（食事）= 撮るだけで献立が決まる |
| ドメイン候補 | `snapmeal.app` / `snapmeal.io`（取得要確認） |

### デザインシステム

| 項目 | 値 |
|---|---|
| Primary | `#FF6B35`（ウォームオレンジ・食欲促進） |
| Surface | `#FEFDF8`（生成り白・温かみ） |
| Text | `#1C1C1E`（オフブラック） |
| Accent | `#2D9E6B`（野菜グリーン・完了・成功状態） |
| Muted | `#8E8E93`（サブテキスト） |
| フォント | `Noto Sans JP`（日本語）+ `Inter`（英語） |
| 角丸 | `rounded-2xl`（やさしい印象） |
| アイコン | Lucide Icons |
| デザイン方針 | 料理画像を大きく見せる・モバイルファースト |

### MVPで検証すること（KPI）

| 検証項目 | 成功指標（仮） |
|---|---|
| 写真アップロードするか | セッションの60%以上で1枚以上 |
| AI提案を見て献立を決定できるか | 「この献立にする」タップ率 40%以上 |
| 「別の献立」ボタンが使われるか | 全セッションの30%以上で使用 |
| 実際に料理するか | ユーザーアンケートで計測 |
| 7日間で複数回使われるか | 7日間リテンション 20%以上 |
| 月額課金に耐える利用頻度か | 週2回以上使用するユーザー割合 |

### 画面遷移の全体像

┌──────▼──────┐
│ ② AI解析中  │  食材スキャン → 献立生成（目標: 10秒以内）

## 3. 考慮ポイント一覧

### 3-1. UX / プロダクト設計

#### 提案体験
- **最初に見せるのは1案だけ**（大量提示はユーザーを疲れさせる）
- 裏側で3案を事前生成し、「別の献立」でstate切り替えのみ（API待ち時間ゼロ）
- 3案使い切った後は「再提案する」で再リクエスト
- 献立カードの情報優先度：料理名 > 調理時間 > 使用食材 > 難易度

#### 食事タイミング
- **朝食・昼食・夕食すべてを対象**（制限なし）
- ユーザーが任意で「今何を作りたいか」を選択できる（選択しなくてもOK）
- プロンプトに食事タイミングを渡し、提案内容を調整（朝は手軽・昼は軽め・夕食はボリューム）

#### 速度設計（最優先）
- **目標: 写真アップロード完了 → 献立表示まで10秒以内**
- クライアント側で画像圧縮（1枚あたり1MB以下）後に送信
- 食材認識と献立3案生成を1回のAPIリクエストに集約
- ストリーミングレスポンスで進捗をリアルタイム表示
- 「別の献立」ボタンはstate変更のみ（追加API呼び出しなし）

#### 写真アップロード
- 枚数上限5枚・明示的なUI
- カメラ直撮り・ギャラリー選択の両方対応
- 撮り直し・削除UI
- アップロード前にブラウザ側で圧縮

#### ローディング体験
- 「食材を認識しています...」→「献立を考えています...」→「もうすぐです!」
- プログレスバーまたはアニメーションで待ち時間を演出

#### 家庭設定
- 人数（大人・子どもの人数）
- アレルギー・苦手食材（チェックリスト＋フリーテキスト）
- 好みのジャンル（和食/洋食/中華/何でも）
- 調理時間の上限（15分/30分/1時間）
- **設定はlocalStorageに保存（認証不要）**

#### エラー体験
- 食材が認識できなかった場合：「もう少し明るい場所で撮影してみてください」
- ネットワークエラー：再試行ボタン
- API障害：「少し混んでいます。しばらくしてから試してください」

### 3-2. 技術スタック

#### フロントエンド
- **Next.js 14+（App Router）** — PWA対応・Vercelとの親和性
- **Tailwind CSS** — モバイルファースト・高速開発
- **@ducanh2912/next-pwa** — PWA化（manifest、service worker）
- **shadcn/ui** — 必要最小限のコンポーネント

#### バックエンド
- **Next.js API Routes** — モノレポで完結、将来的に切り出し可能

#### AI / Vision（後述のAPI比較セクション参照）
- **推奨: Gemini 2.5 Flash**（コスト・品質・速度のバランスが最優）
- 画像はbase64エンコードしてAPIに送信（サーバー保存なし）
- ストリーミングAPIで食材認識結果を先出し、体感速度を改善
- 将来的に品質不足なら GPT-4.1 / GPT-4o にアップグレード

#### ストレージ・DB
- **MVPはDB不要**
- 家庭設定: `localStorage`
- 生成した献立3案: React state（セッション内のみ）
- 画像: メモリのみ（サーバー保存なし）

#### 多言語対応（i18n）
- **最初から多言語構造を設計**（日本語先行、英語を次フェーズ）
- `next-intl` を使用
- 翻訳ファイル: `messages/ja.json` / `messages/en.json`
- AIの出力言語はユーザーのロケールに合わせてプロンプトで制御
- URLルーティング: `/ja/...` / `/en/...`（またはブラウザ言語自動検出）

#### 認証
- **MVPは認証なし**
- ただし課金モデル実装時は認証が必須になるため、next-authを後から追加できる構造にしておく

#### デプロイ
- **Vercel**（無料枠でMVP開始）
- 環境変数: `ANTHROPIC_API_KEY`, `STRIPE_SECRET_KEY`（後から追加）

### 3-3. AI プロンプト設計（品質の核心）

#### 1リクエストで完結する設計
食材認識と献立3案生成を分けず、**1回のAPIコールで完結**させる。

```
System: あなたは家庭料理の専門家です。
        ユーザー設定: {家族構成・アレルギー・調理時間・食事タイミング}
        言語: {ja/en}

User: [画像1〜5枚] + 「冷蔵庫の写真から献立を3案提案してください。JSON形式で返してください。」
```

#### 献立生成の方針
- 認識された食材を**できるだけ消費**する方向で提案
- 食事タイミング（昼食/夕食）に応じたボリューム調整
- 足りない食材は「買い物リスト」として明示
- 3案は**バラエティを意図的に持たせる**（和・洋・中 など）

#### 出力スキーマ
```json
{
  "detected_ingredients": ["豚バラ", "大根", "卵", "しょうゆ", "みりん"],
  "meals": [
    {
      "id": "1",
      "name": "豚バラ大根",
      "time_minutes": 30,
      "difficulty": "easy",
      "description": "冷蔵庫の大根と豚バラを使った定番煮物",
      "matched_ingredients": ["豚バラ", "大根", "しょうゆ", "みりん"],
      "missing_ingredients": [],
      "recipe": {
        "servings": 4,
        "ingredients": [{"name": "豚バラ", "amount": "300g"}],
        "steps": ["豚バラを一口大に切る", "大根を...", "..."]
      }
    }
  ]
}
```

#### APIコスト見積もり（Gemini 2.5 Flash 採用時）
| 項目 | 数値 |
|---|---|
| Input (画像5枚+プロンプト) | 約18,450トークン → $0.0055 |
| Output (献立3案) | 約3,000トークン → $0.0075 |
| **1セッション合計** | **約$0.013（約2円）** |
| 月30セッション(Paid) | 約$0.39（60円）|
| 月5セッション(Free) | 約$0.065（10円・CAC扱い）|

→ **粗利率は約96%**（当初目標80%を大きく上回る）

### 3-5. プライバシー・法的考慮

- **個人開発（個人事業主）** として運営
- プライバシーポリシーに必ず明記：「写真はAnthropicのAPIに送信後、当社サーバーに保存しない」
- アレルギー情報の免責：「本アプリは医療的助言を行いません」
- Anthropic利用規約の確認（商用利用・学習利用の有無）
- **TermsFeed**（$9〜）または **iubenda** で規約を作成
- リリース前に利用規約・プライバシーポリシーのページを必ず設置

---

### 3-6. パフォーマンス

- 画像圧縮: `browser-image-compression` でアップロード前に1MB以下に圧縮
- 献立3案を1リクエストで生成（ラウンドトリップ削減）
- ストリーミングAPIで結果を段階的に表示
- 「別の献立」はstateのindex変更のみ（即時）
- Service WorkerでPWA静的アセットをキャッシュ
- Vercel Edge Functionsの活用検討（低レイテンシ）

---

### 3-7. モバイル最適化（PWA）

- `manifest.json`: テーマカラー`#FF6B35`・display: standalone
- カメラアクセス: `<input type="file" accept="image/*" capture="environment">`
- タッチ操作: ボタンサイズ44px以上
- iOS Safariのホーム画面追加を手動で案内するUI
- OGP設定（SNSシェア時のプレビュー）
- iOS Safariの制約（ServiceWorker・FileAPI）を事前検証

---

### 3-8. スコープ外（MVPに含めない）

以下は意図的に除外し、KPI検証後に判断する：

- ユーザー認証・ログイン（Stripe課金連携時に追加）
- 献立履歴の保存
- お気に入り登録
- SNSシェア機能
- 栄養計算・カロリー表示
- 週次献立プランニング
- 食材の在庫管理（継続トラッキング）
- プッシュ通知
- ネイティブアプリ（iOS/Android）
- 英語以外の言語（日本語・英語のみでMVP）

## 5. リスクと対策

| リスク | 影響度 | 対策 |
|---|---|---|
| AI食材認識精度が低い | 高 | プロンプト改善・複数枚推奨・ユーザーフィードバックUI |
| API応答が遅い（>15秒） | 高 | ストリーミングAPI＋クライアント圧縮で目標10秒以内 |
| 粗利80%を下回る | 高 | プロンプト最適化でトークン削減・使用上限厳守 |
| iOS Safariの制約 | 中 | PWA非依存の設計・事前実機検証 |
| プライバシー懸念でDL拒否 | 中 | 「写真を保存しない」を最初の画面で訴求 |
| 献立提案がマンネリ化 | 低 | プロンプトに多様性パラメータ追加（seed変更） |

## 8. API選定比較（Vision + レシピ生成）

### 用途別タスク
1. **画像解析**: 冷蔵庫写真（最大5枚）から食材を認識
2. **献立生成**: 認識した食材と家庭設定から献立3案をJSON出力

両タスクを**1回のAPIリクエストで完結**させる設計。

### コスト比較（1セッション = 画像5枚 + 入力12,000tok + 出力3,000tok）

| モデル | 1セッション | 月30回(Paid) | 粗利率 | Vision精度 | 日本語品質 |
|---|---|---|---|---|---|
| **Gemini 2.5 Flash** ⭐推奨 | $0.013 | $0.39 | **96%** | 中〜高 | 中〜高 |
| GPT-4o-mini | $0.004 | $0.12 | **98%** | 中 | 中 |
| GPT-4.1-mini | $0.012 | $0.36 | **96%** | 中 | 中 |
| Gemini 2.5 Pro | $0.053 | $1.59 | **84%** | 高 | 高 |
| GPT-4.1 | $0.058 | $1.74 | **81%** | 高 | 高 |
| GPT-4o | $0.072 | $2.16 | **77%** ✗ | 高 | 高 |
| Claude Haiku 4.5 | $0.041 | $1.23 | **87%** | 中 | 中 |
| Claude Sonnet 4.6 | $0.122 | $3.66 | **61%** ✗ | 高 | 高 |
| Ollama (ローカル) | $0 | $0 | **〜100%** | 低〜中 | **低** ✗ |

> ※ Claude系は画像を `(幅×高さ)/750` トークンで計算するため画像コストが高い。  
> ※ Ollama/ローカルLLMは日本語Vision対応が公式サポート外のため品質保証困難。  
> ※ Gemini 2.0 Flash・1.5 Flash は廃止済み／廃止予定のため選択肢外。

### 推奨構成

#### MVP（Day 1）
**Gemini 2.5 Flash** を採用

- 粗利96%で目標80%を大幅上回る
- 無料枠あり（開発・テストコスト削減）
- Vision精度・日本語品質ともに十分
- Google AI SDK（`@google/generative-ai`）でNext.jsに統合容易

#### 品質問題が発生した場合のエスカレーション
```
Gemini 2.5 Flash → GPT-4.1 → GPT-4o
```
GPT-4.1 採用でも粗利81%、80%ラインをキープ可能。

#### 将来のコスト最適化（ユーザー数が増えたら）
- 食材認識: **Gemini 2.5 Flash**（安い・速い）
- レシピ生成のみ: **GPT-4.1-mini** または **Claude Haiku**
- 2段階APIに分けることでさらにコスト圧縮可能（ただしレイテンシが増えるためMVPでは非推奨）
