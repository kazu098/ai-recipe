## 1. プロダクト定義

### 1-1. 本質

これは**レシピ検索アプリではない**。

**共働きで子どもがいる家庭の親が、平日18-19時の「今夜の夕食を何にするか」という意思決定疲れから解放されるためのAIオペレーションツール**である。

ユーザーが欲しいのは大量のレシピではなく、「今日これを作ればいい」と**決めてもらう体験**。

### 1-2. ターゲットユーザー（プライマリペルソナ）

**「共働きで子どもがいる家庭の親」** をグローバル共通のコアペルソナとして定義。  
ローンチは日本から、英語版をすぐ後追いで投入する。

| 属性 | 内容 |
|---|---|
| 世帯構成 | 共働き夫婦＋子ども1〜2人（3〜10歳） |
| 居住地 | グローバル（日本ローンチ→英語圏→欧州→アジア） |
| 平日の課題 | 17:30-18:30の「今夜何作ろう」パニック・買い物の二度手間 |
| 最大の制約 | **時間**（お金ではない） |
| 価格感度 | 低い（時短サービスに月$6.49 / ¥980は許容） |
| ITリテラシー | 中（メッセンジャー・モバイル決済・ECは使える） |
| デバイス | iPhone 6〜7割 / Android 3〜4割（地域差あり） |
| 利用シーン | 平日18時前後 / 週末まとめ買い後 |

### 1-3. North Star Metric

**WAFC (Weekly Active Family Cooks)** = 7日間に1回以上「この献立で作る」をタップしたユニーク世帯数

サブメトリクス：
- **W1リテンション**: 初回利用から7日以内の再利用率（目標30%）
- **平日夕方の利用集中度**: 17-19時利用が全体に占める割合（目標50%以上）
- **Pro転換率**: Freeユーザー → Pro月額（目標5%）
- **粗利率**: AI+決済手数料控除後で90%以上（Gemini 2.5 Flash-Lite採用）

**B2B優先度と戦略:**

| 優先度 | 相手 | 価値提案 | 収益モデル |
|---|---|---|---|
| ① | **シャープ（ホットクック）** | ユーザーの使いこなし支援 → 解約率低下・満足度向上 | レベニューシェア or 機能ライセンス |

**B2BはB2Cで1万ユーザー超・データ蓄積後に提案する。それ以前は提案資料が弱い。**

**価格の正当化:**  
共働き夫婦が「今夜何作ろう」と悩む時間は平日1日5分 × 20日 = 月100分。  
時給¥1,500換算で月¥2,500相当の時間コストを¥980で代替。**コスパ3倍以上**。  
Cookpadとの比較は不適切——Cookpadは「自分でレシピを選ぶ道具」、Snapmealは「決めてもらうサービス」で別カテゴリ。

### 2-5. 長期ビジョン：家庭の食OSへの拡張

「冷蔵庫を撮って献立を決める」はあくまで**第一接点**。本質的には「家庭の食にまつわる意思決定を代行するOS」を目指す。

```
[買い物] → [冷蔵庫管理] → [献立決定] → [調理サポート] → [食事記録] → [振り返り]
             賞味期限管理      ★今ここ      将来            月次レポート   栄養バランス
             写真認識OCR
```

**B2C縦拡張のロードマップ（優先順）:**

| 機能 | Phase | なぜ重要か |
|---|---|---|
| **使い切り食材のオプトイン指定** | 2 | 食材確認画面でタップするだけ。賞味期限管理より軽く、ユーザー意図が直接反映される |
| **家族プロフィール深化**（好き嫌い学習） | 2 | 使うほど乗り換えコスト上昇。データmoatの核 |
| **食事記録 → 月次レポート** | 3 | 感情的価値大（子どもに何を食べさせたか）→ 解約防止 |
| **調理サポート（音声）** | 3 | 調理工程へ踏み込む。競合との差別化 |
| **栄養バランス可視化** | 4 | 健康ジャンルへ越境。ARPU引き上げ |

**天井を壊す設計原則：「決定支援」を絶対に軸として維持し、その前後のジャーニーをジョイントする。レシピ検索アプリ化（≒クックパッド化）は行わない。**

### 2-6. 無料 → Pro 転換設計

- **ゲスト3回到達時**にログイン促進（無料アカウント作成で月10回へ）
- **お気に入り保存時**にログイン促進（保存価値を理由に自然に登録へ誘導）
- **Free 10回/月到達時**にProアップグレードのハードブロック
- Pro訴求は「月90回まで」「回数を気にせず使える」を前面に出す
- Annual/初月割引は将来施策。MVPでは Monthly を先に安定させる

```
      │

│  ④ レシピ詳細（材料・手順・買い物） │
└────────────────────────────────────┘
      │
      ├─ お気に入り保存 → 未ログインならログイン促進
      ├─ ゲスト3回到達 → ログイン促進
      └─ Free 10回/月到達 → Proアップグレード
```

**ログインタイミング（現行方針）:**
- 初回起動時にログインを強制しない。まずオンボーディング後にゲストで価値体験させる
- ゲストは端末ローカルカウントで3回まで解析可能
- ログイン方法はメール+パスワード / Google OAuth。マジックリンクは現行MVPでは主導線にしない
- ログイン後は `profiles` 行を自動作成し、月次利用回数を `usage_counters` で管理する

**課金タイミング（現行方針）:**
- 認証済みFreeユーザーが月10回に達したら、解析APIが `usage_limit_exceeded` を返す
- クライアントはアップグレードモーダルを表示し、Stripe Checkoutへ遷移する
- Checkout完了後、Stripe webhookで `profiles.plan = 'pro'` に更新する

### 4-4. 画像処理（Vercel上限対策）

**想定枚数: 推奨3枚**（冷蔵庫の正面・野菜室・冷凍庫を想定）。最大5枚まで許容。

問題: Vercel Serverless関数のペイロード上限 4.5MB → 5枚×1MB で超過リスク

**MVPの現行解決策:**
1. クライアントで画像を圧縮（長辺1280px、JPEG品質0.7）
2. 圧縮後の `imageDataUrls` を `/api/analyze` に送信
3. サーバー側で画像を1枚ずつGemini APIに並列送信（per-image認識）
4. 画像はDB/Storageに保存しない

これにより：
- 推奨3枚の圧縮済み画像はペイロード約1.5MB → Vercel上限の余裕あり
- 1枚ずつ送信するため、枚数が増えてもAPI側の処理は安定

**将来方針:** 画像改善オプトインを実装する段階で、Supabase Storageへの直接アップロードに移行する。

### 4-5. ストリーミング設計（3フェーズ）

**Phase A-1（画像認識・並列）→ Phase A-2（献立生成・テキスト）→ Phase B（代替案・バックグラウンド）**

```typescript
// Phase A-1: 画像を1枚ずつ並列認識
const lists = await Promise.all(images.map(recognizeIngredientsOneImage));
const ingredients = mergeIngredients(lists);
// → ingredient イベントを逐次配信

// Phase A-2: 認識済み食材リスト（テキスト）のみで献立生成
const meal = await generateMealFromIngredients(ingredients, ...settings);
// → meal イベントを配信
```

**Phase A-1（画像認識・並列）- ユーザー体験:**
- 0〜5秒: 画像を1枚ずつ並列認識（3枚なら3並列）
- 食材名が次々ストリーミング表示（`ingredient` イベント）
- 各画像の認識結果をマージ・重複除去して確定

**Phase A-2（献立生成・テキストのみ）:**
- 認識済み食材リストのみをGeminiへ送信（画像は再送しない）
- 目標: A-1完了から2〜3秒で献立完成 → **即遷移**（合計目標8秒以内）

**Phase B（代替案・バックグラウンド）:**
- Phase Aの `session` イベント受信と同時にクライアントが自動呼び出し
- 食材リスト（テキスト）のみをGeminiへ送信 → Vision処理不要で高速・安価
- 目標: 3〜5秒で完了（ユーザーが1案目を読んでいる間に準備完了）
- 「別の献立を見る」タップ時に即表示（待ち時間ゼロ）

**コスト削減効果:**  
Phase A-1 で軽量なJSON配列（食材名のみ）を返すため出力トークンが最小。  
Phase A-2 以降は画像不要のテキスト処理のみ。**セッション全体を通じて画像トークンは3回分のみ**（従来の全一括送信と同等コスト・精度は大幅向上）。

| 用途 | 入力 | 出力 | 1回あたり |
|---|---|---|---|
| レシピ生成（Flash-Lite） | 約400トークン | 約600トークン | **≒$0.0003（約0.04円）** |

**旧方式との比較:**

| 方式 | Vision呼び出し | 合計/セッション（Flash-Lite） | 精度 |
|---|---|---|---|
| 旧: 全画像を一括送信 | 1回（複数画像） | $0.005 | △ 画像多いほど精度低下 |
| **新: per-image並列認識** | **3回（1枚×3並列）** | **$0.003** | **✅ 各画像を正確に認識** |

per-image方式は**コストも約40%削減**しつつ精度が向上。

※ 推奨3枚・1024px・JPEG品質0.7の圧縮画像を想定。Vision料金はGemini公式のトークン換算（約258token/枚）で概算。
⚠️ **コストの正確な現行価格は公式で確認すること:**  
- OpenAI: `platform.openai.com/docs/pricing`  
- Google Gemini: `ai.google.dev/gemini-api/docs/pricing`

**粗利率の補足:**  
Gemini 2.5 Flash-Lite Primary採用により、Pro月額¥980で30回利用した場合のAIコストは月$0.09（約¥14）。  
旧方式の$0.15（約¥22）から**さらに約35%削減**。  
Stripe国内カード手数料3.6%を含めても、30〜90回/月の範囲で粗利90%超を維持できる見込み。GPT-4oフォールバック率が上がると粗利に直接効くため、フォールバック率を計測対象にする。

### 4-9. ホットクック対応方針

ホットクックは公式メニュー番号の完全DBではなく、**カテゴリ + 調理原理エンジン**で扱う。目的は「実機で再現しやすい料理だけを提案し、メニューの探し方と手動フォールバックを提示する」こと。

**対応カテゴリ:**
- 煮物（肉/魚/野菜/乾物）
- カレー・シチュー
- スープ
- 蒸し物
- 無水ゆで
- 発酵・低温調理
- ごはん/リゾット
- 炒め煮（しっとり系）

**非対応として避ける料理:**
- 揚げ物、焦げ目が必要な焼き物、シャキッと炒め、焼き卵料理、パン/グリル、生もの
- ユーザーが非対応料理をリクエストした場合は、近い対応料理に置換する

**レシピ詳細で表示する内容:**
- 推奨カテゴリ名と説明
- 実機での操作経路
- 自動メニュー例
- 手動フォールバック（モード/沸とう後時間）
- 水分、まぜ技、時間、容量、下処理、安全注意

## 10. Phase 2 機能（MVPリリース後・リテンション強化）

「目の前の食材で今何を作るか」というコア哲学に沿ったリテンション施策を Phase 2 で順次投入。

### 10-1. お気に入り機能（最優先）
- 献立カードに ♡ ボタン
- 「お気に入り」一覧画面
- 次回の提案時、お気に入り傾向をプロンプトに含める
  → AI が「うちの好み」を学習する第一歩
- DBスキーマ拡張: `favorites` テーブル追加

### 10-2. AI嗜好学習（差別化・moat構築）
ユーザーの「作った！」「別の献立にした」フィードバックを蓄積し、提案精度を継続改善。

**プロンプトへの注入内容:**
- お気に入り料理の傾向（甘め/辛め/魚多め/肉多め）
- 「作った！」率の高い料理タイプ
- 「別の献立を見る」で却下されやすい料理タイプ
- 平日/週末の傾向差

**効果:** 使うほど自分仕様になる → 解約コスト上昇。データmoatの中核。

### 10-3. 買い物提案＋アフィリエイト（収益moat）
レシピ詳細の「買い物リスト」から、不足食材を即購入できる導線。

```
[🛒 買い物リスト]
  ├─ ケチャップ
  │  [🛍 Amazonで買う]    ← アフィリエイトリンク（初期）
  │  [🛍 楽天で買う]
  ├─ にんじん
  │  [🚚 Oisixで明日届く]  ← 提携収益（将来）
  │  [📍 近隣スーパーを探す]  ← 将来
```

**収益試算:**  
1万ユーザー × 30%が利用 × 月¥3,000買物 × 5%手数料 = **月¥45万 / ARR ¥540万**  
サブスクと並ぶ収益の柱に育成可能。

**実装フェーズ:**
- Phase 2a: Amazon アフィリエイト（個人開発でも即取得可）
- Phase 2b: 楽天アフィリエイト
- Phase 3: Oisix・パルシステム等とのB2B提携

### 10-4. 冷蔵庫整理スコア（食品ロス削減・サステナブル軸）
共働き親に強く響く「食品ロス削減」を可視化。

```
[今日の食材消費率]
░░░░░░░░░░░░░░░░ 87%

[今月の累計]
推定削減量: 2.1kg
節約金額: ¥1,840

[今年の累計]
削減量: 18.4kg ≒ 二酸化炭素換算 36.8kg削減 🌱
```

- 提案献立の `matched_ingredients` から自動計算
- 月次サマリーをメールで送信 → **再エンゲージの動機にもなる**
- SDGs意識との接続でメディア露出にも貢献

### 10-5. 使い切り食材のオプトイン指定（食材確認画面）

賞味期限の日付管理は行わない。代わりに**「今日使いたい食材をタップして優先指定できる」**シンプルなUXを実装。

**設計方針:**
- 食材認識完了後、食材確認画面で止まる（自動遷移なし・カウントダウンなし）
- 食材チップをタップ → ハイライト → 優先使用フラグが立つ（オプトイン）
- 何もタップせず「この食材で献立を決める」を押せば全食材から提案（従来と同じ体験）
- タップされた食材はプロンプトに「必ず使うこと」として注入される

**ユーザーの典型的な使い方:**
- もやしや葉物など傷みやすい食材があるとき → それだけタップして優先
- 何も気にしない日 → そのままボタンを押すだけ

**実装コスト:** 低。既存の食材リスト表示にタップ選択UIを追加し、選択結果を `/api/analyze` の `priority_ingredients` パラメータとして渡すだけ。

### 10-6. 食事記録 → 月次レポート（リテンション強化）

毎月1日に「先月の食卓レポート」をメールで配信。

```
📊 4月の食卓まとめ

作った献立: 18回
お気に入り度 1位: 鶏そぼろ丼（3回）
節約できた食材ロス: 推定 1.8kg

家族の食卓、見守っています 🌿
```

- 感情的価値（子どもに何を食べさせてきたか）が退会ブレーキになる
- 月次レポートをReasendで配信 → メールチャンネル確立 → 再エンゲージ導線にも使える
