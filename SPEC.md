## 1. プロダクト定義

### 1-1. 本質

これは**レシピ検索アプリではない**。

**共働きで子どもがいる家庭の親が、平日18-19時の「今夜の夕食を何にするか」という意思決定疲れから解放されるためのAIオペレーションツール**である。

ユーザーが欲しいのは大量のレシピではなく、「今日これを作ればいい」と**決めてもらう体験**。

### 1-2. ターゲットユーザー（プライマリペルソナ）

**「共働きで子どもがいる家庭の親」** をグローバル共通のコアペルソナとして定義。  
ローンチは日本から、英語版をすぐ後追いで投入する。

| 属性 | 内容 |
|---|---|
| 世帯構成 | 共働き夫婦＋子ども1〜2人（3〜10歳） |
| 居住地 | グローバル（日本ローンチ→英語圏→欧州→アジア） |
| 平日の課題 | 17:30-18:30の「今夜何作ろう」パニック・買い物の二度手間 |
| 最大の制約 | **時間**（お金ではない） |
| 価格感度 | 低い（時短サービスに月$9.99 / ¥980は許容） |
| ITリテラシー | 中（メッセンジャー・モバイル決済・ECは使える） |
| デバイス | iPhone 6〜7割 / Android 3〜4割（地域差あり） |
| 利用シーン | 平日18時前後 / 週末まとめ買い後 |

### 1-3. North Star Metric

**WAFC (Weekly Active Family Cooks)** = 7日間に1回以上「この献立で作る」をタップしたユニーク世帯数

サブメトリクス：
- **W1リテンション**: 初回利用から7日以内の再利用率（目標30%）
- **平日夕方の利用集中度**: 17-19時利用が全体に占める割合（目標50%以上）
- **Pro転換率**: Freeユーザー → Pro月額（目標5%）
- **粗利率**: 80%以上（Gemini採用で実質94%）

### 4-2. データモデル（Supabase）

```sql
-- ユーザー（Supabase Authに紐づく）
profiles (
  id UUID PRIMARY KEY,             -- auth.users.id
  email TEXT NOT NULL,
  created_at TIMESTAMPTZ,
  household_settings JSONB,         -- 家庭設定をまとめて
  plan TEXT DEFAULT 'free',         -- 'free' / 'pro' / 'pro_annual'
  stripe_customer_id TEXT,
  locale TEXT DEFAULT 'ja',
  photo_optin BOOLEAN DEFAULT false -- 品質改善に画像提供OK
);

-- セッション（1回の解析リクエスト）
sessions (
  id UUID PRIMARY KEY,
  user_id UUID REFERENCES profiles(id),
  created_at TIMESTAMPTZ,
  tired_mode BOOLEAN,               -- 「疲れた」選択フラグ
  detected_ingredients JSONB,
  meals JSONB,                      -- 3案（phase_a: 1案, phase_b: 2案）
  selected_meal_id TEXT,            -- どれを選んだか
  cooked BOOLEAN DEFAULT false,     -- 「作った！」フィードバック
  storage_paths TEXT[]              -- 画像（optin時のみ）
);

-- マンネリ回避用：提案された料理の履歴（MVPから必須）
meal_history (
  id UUID PRIMARY KEY,
  user_id UUID REFERENCES profiles(id),
  session_id UUID REFERENCES sessions(id),
  meal_name TEXT NOT NULL,
  genre TEXT,                       -- 和食/洋食/中華/エスニック
  main_ingredient TEXT,             -- 肉/魚/卵/野菜/麺/米
  cooking_method TEXT,              -- 炒め/煮込み/焼き/揚げ/蒸し/サラダ
  was_selected BOOLEAN DEFAULT false, -- 「この献立で作る」されたか
  was_cooked BOOLEAN DEFAULT false,   -- 「作った！」されたか
  created_at TIMESTAMPTZ
);
-- INDEX: (user_id, created_at DESC) で直近の履歴を高速取得

-- 月次使用量
usage_counters (
  user_id UUID REFERENCES profiles(id),
  year_month TEXT,                  -- '2026-05'
  count INT DEFAULT 0,
  PRIMARY KEY (user_id, year_month)
);
```

**RLS（Row Level Security）**: 各テーブルで `auth.uid() = user_id` のみ閲覧可。

### 4-3. API設計

```
── Phase A: 画像解析 + 1案目即時生成 ──────────────────────────

POST /api/analyze
  body: { storage_urls: string[], tired_mode: boolean, meal_time?: 'morning'|'lunch'|'dinner' }
  → Server-Sent Events
     1. { event: 'ingredients', data: string[] }         ← 食材リスト（ここでUIに食材表示）
     2. { event: 'meal_1', data: { id, type, name, ... } } ← 1案目完成 → 即画面遷移
     3. { event: 'session', data: { session_id } }       ← セッションID（Phase B呼び出しに使用）
     4. { event: 'done' }
  認証: Supabase JWT必須
  上限チェック: usage_counters 参照
  目標レスポンス: 食材表示まで3秒 / 1案目表示まで8秒

── Phase B: テキストのみで残り2案をバックグラウンド生成 ─────────

POST /api/sessions/{id}/alternatives
  body: { ingredients: string[], tired_mode: boolean }   ← 画像なし・食材リストのみ
  → Server-Sent Events
     1. { event: 'meal_2', data: { id, type, name, ... } }
     2. { event: 'meal_3', data: { id, type, name, ... } }
     3. { event: 'done' }
  タイミング: Phase Aの 'session' イベント受信直後にクライアントが自動呼び出し
  目標: ユーザーが1案目を読んでいる間（3〜5秒）に完了

── 共通 ────────────────────────────────────────────────────────

POST /api/sessions/{id}/select
  body: { meal_id }
  → 選んだ献立を記録（KPI計測用）

POST /api/sessions/{id}/cooked
  → 「作った！」フィードバック記録（重要KPI）

POST /api/webhooks/stripe
  → 課金状態の同期
```

**B2B提携前提のAPI設計（将来）:**
- `POST /api/v1/partner/analyze` — APIキー認証・パートナー用
- 食材リストを直接受け取るエンドポイント（Oisix連携想定）

### 4-4. 画像処理（Vercel上限対策）

問題: Vercel Serverless関数のペイロード上限 4.5MB → 5MB（5枚×1MB）で超過

**解決策:**
1. クライアントで画像圧縮後、**Supabase Storage に直接アップロード**
2. ストレージURLを `/api/analyze` に渡す
3. サーバー側でURLからGemini APIに送信
4. 解析完了後、optin=falseなら削除

これにより：
- Vercel関数のペイロード問題回避
- optin時の画像保存も同じ仕組みで完結

### 4-5. ストリーミング設計（2フェーズ）

Gemini APIの`generateContentStream`を使い、出力JSONをパース可能な単位で配信：

```typescript
for await (const chunk of stream) {
  const parsed = parseStreamingJSON(chunk);
  if (parsed.event) {
    res.write(`data: ${JSON.stringify(parsed)}\n\n`);
  }
}
```

**Phase A（画像あり・1案目）- ユーザー体験:**
- 0-3秒: 「食材スキャン中」アニメーション
- 3-6秒: 食材名が次々ストリーミング表示
- 6-8秒: 1案目の献立が完成 → **即遷移**（目標：8秒以内）

**Phase B（画像なし・残り2案）- バックグラウンド:**
- Phase Aの `session` イベント受信と同時にクライアントが自動呼び出し
- 食材リスト（テキスト）のみをGeminiへ送信 → Vision処理不要で高速・安価
- 目標: 3〜5秒で完了（ユーザーが1案目を読んでいる間に準備完了）
- 「別の献立を見る」タップ時に即表示（待ち時間ゼロ）

**コスト削減効果:**  
Phase Aで画像トークンを消費後、Phase Bはテキストのみ。  
5案を画像付きで生成する従来方式に比べ、**画像トークン2〜4枚分削減**（≒ 40%コスト削減）。

ユーザーが「待った」と感じる時間はストリーミング開始までの数秒であり、合計16秒は問題にならない。
