## 1. プロダクト定義

### 1-1. 本質

これは**レシピ検索アプリではない**。

**共働きで子どもがいる家庭の親が、平日18-19時の「今夜の夕食を何にするか」という意思決定疲れから解放されるためのAIオペレーションツール**である。

ユーザーが欲しいのは大量のレシピではなく、「今日これを作ればいい」と**決めてもらう体験**。

### 1-2. ターゲットユーザー（プライマリペルソナ）

**「共働きで子どもがいる家庭の親」** をグローバル共通のコアペルソナとして定義。  
ローンチは日本から、英語版をすぐ後追いで投入する。

| 属性 | 内容 |
|---|---|
| 世帯構成 | 共働き夫婦＋子ども1〜2人（3〜10歳） |
| 居住地 | グローバル（日本ローンチ→英語圏→欧州→アジア） |
| 平日の課題 | 17:30-18:30の「今夜何作ろう」パニック・買い物の二度手間 |
| 最大の制約 | **時間**（お金ではない） |
| 価格感度 | 低い（時短サービスに月$9.99 / ¥980は許容） |
| ITリテラシー | 中（メッセンジャー・モバイル決済・ECは使える） |
| デバイス | iPhone 6〜7割 / Android 3〜4割（地域差あり） |
| 利用シーン | 平日18時前後 / 週末まとめ買い後 |

### 1-3. North Star Metric

**WAFC (Weekly Active Family Cooks)** = 7日間に1回以上「この献立で作る」をタップしたユニーク世帯数

サブメトリクス：
- **W1リテンション**: 初回利用から7日以内の再利用率（目標30%）
- **平日夕方の利用集中度**: 17-19時利用が全体に占める割合（目標50%以上）
- **Pro転換率**: Freeユーザー → Pro月額（目標5%）
- **粗利率**: 80%以上（Gemini採用で実質94%）

### 4-2. データモデル（Supabase）

```sql
-- ユーザー（Supabase Authに紐づく）
profiles (
  id UUID PRIMARY KEY,             -- auth.users.id
  email TEXT NOT NULL,
  created_at TIMESTAMPTZ,
  household_settings JSONB,         -- 家庭設定をまとめて
  plan TEXT DEFAULT 'free',         -- 'free' / 'pro' / 'pro_annual'
  stripe_customer_id TEXT,
  locale TEXT DEFAULT 'ja',
  photo_optin BOOLEAN DEFAULT false -- 品質改善に画像提供OK
);

-- セッション（1回の解析リクエスト）
sessions (
  id UUID PRIMARY KEY,
  user_id UUID REFERENCES profiles(id),
  created_at TIMESTAMPTZ,
  tired_mode BOOLEAN,               -- 「疲れた」選択フラグ
  detected_ingredients JSONB,
  meals JSONB,                      -- 5案
  selected_meal_id TEXT,            -- どれを選んだか
  cooked BOOLEAN DEFAULT false,     -- 「作った！」フィードバック
  storage_paths TEXT[]              -- 画像（optin時のみ）
);

-- 月次使用量
usage_counters (
  user_id UUID REFERENCES profiles(id),
  year_month TEXT,                  -- '2026-05'
  count INT DEFAULT 0,
  PRIMARY KEY (user_id, year_month)
);
```

**RLS（Row Level Security）**: 各テーブルで `auth.uid() = user_id` のみ閲覧可。

### 4-3. API設計


### 4-4. 画像処理（Vercel上限対策）

問題: Vercel Serverless関数のペイロード上限 4.5MB → 5MB（5枚×1MB）で超過

**解決策:**
1. クライアントで画像圧縮後、**Supabase Storage に直接アップロード**
2. ストレージURLを `/api/analyze` に渡す
3. サーバー側でURLからGemini APIに送信
4. 解析完了後、optin=falseなら削除

これにより：
- Vercel関数のペイロード問題回避
- optin時の画像保存も同じ仕組みで完結

### 4-5. ストリーミング設計

Gemini APIの`generateContentStream`を使い、出力JSONをパース可能な単位で配信：

```typescript
for await (const chunk of stream) {
  const parsed = parseStreamingJSON(chunk);
  if (parsed.event) {
    res.write(`data: ${JSON.stringify(parsed)}\n\n`);
  }
}
```

ユーザー体験：
- 1-3秒目: 「食材スキャン中」（ストリーミング前）
- 3-6秒目: 食材名が次々表示
- 6-10秒目: 1案目の献立が完成 → **即遷移**

### 4-7. AI / プロンプト設計

**1リクエストで完結:** 食材認識と5案生成を分離せず、1コールで処理。

```
System Prompt:
あなたは家庭料理の専門家です。日本の共働き家庭向けに献立を提案してください。

ユーザー設定:
- 大人: {adults}人 / 子ども: {kids_ages}
- アレルギー: {allergies}
- NG食材: {ng_foods}
- 調理器具: {appliances}
- 味の好み: {taste}
- 料理方針: {policy_memo}
- 言語: ja

今日の状況:
- 食事: {meal_time}
- 余力: {tired_mode ? "疲れている。15分以内で作れるものを優先" : "通常"}

出力: 必ず以下JSONスキーマで5案。
- type: best / quick / no_shopping / kids_friendly / use_up を1つずつ
- バラエティ: 和洋中バランス・調理法バランス
- 子ども向けは薄味・辛味なし
- 賞味期限が短そうな食材を優先消費

[画像5枚]
```

**モデル戦略（実証実績に基づく）:**

> **検証済みベースライン:** 創業者が ChatGPT カスタムGPT（GPT-4o 系 Vision）で冷蔵庫写真の認識を実機検証し、献立提案まで実用品質を確認済み。**この品質を最低ラインとし、それを下回らないモデルを採用する。**

採用順序:
1. **リリース初週〜1ヶ月**: **GPT-4o**（検証済み品質を直接踏襲・$0.072/session・粗利77%）  
   または **Gemini 2.5 Pro**（コスト優位・$0.053/session・粗利84%・要A/B比較）
2. **品質安定・データ蓄積後**: **Gemini 2.5 Flash** にダウングレード検証（$0.018/session・粗利94%）
3. **判定基準**: 内部評価セット30枚で「食材認識F1 ≧ 0.80」「献立妥当性 ≧ 4.0/5.0」をクリアすれば切替

**推奨アクション:** 実装初日に GPT-4o と Gemini 2.5 Pro の **両方を同一プロンプト・同一画像で並列実行**し、創業者の手元で比較できる管理画面を作る。
